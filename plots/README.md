# Benchmark plots

All throughputs are **chip-wide** (per-pod × 8 pods, since all pods
share one DRAM connection). Data sources:

- `results/results_20260423_233526_chipwide.csv` — fast-clock sweep
- `results/results_20260424_072636_sim32bw_chipwide.csv` — slow-clock
  sweep scaled × 32, i.e. projected onto a hypothetical fast-clock
  machine with 32× more DRAM bandwidth

Generated by `generate_plots.py`. Plots saved at 300 dpi.

Chip area assumed: 38.875 mm² (all 8 pods together).


## Single-algorithm scaling

### `01_sw1d_gcups_vs_seqlen`

![01_sw1d_gcups_vs_seqlen](01_sw1d_gcups_vs_seqlen.png)

sw/1d chip-wide GCUPS at cpg=8, shared-data config. Shows the throughput ramp from short sequences (startup-dominated) to the ~31 GCUPS plateau around seq_len=256+.

### `02_sw1d_time_vs_seqlen`

![02_sw1d_time_vs_seqlen](02_sw1d_time_vs_seqlen.png)

sw/1d per-alignment wallclock on log-log axes. Follows the O(n²) reference slope closely, with a slight efficiency gain at larger sizes (the pipeline-fill overhead amortizes).

### `03_sw2d_gcups_vs_seqlen`

![03_sw2d_gcups_vs_seqlen](03_sw2d_gcups_vs_seqlen.png)

sw/2d chip-wide GCUPS. Sweeps only four sizes (32, 64, 128, 192) because the 2D systolic DMEM footprint is dominant.

### `04_sw2d_time_vs_seqlen`

![04_sw2d_time_vs_seqlen](04_sw2d_time_vs_seqlen.png)

sw/2d per-alignment wallclock on log-log axes. Also O(n²) but with a steeper constant factor at small sizes relative to sw/1d.

## Algorithm comparison

### `05_algos_gcups_vs_seqlen`

![05_algos_gcups_vs_seqlen](05_algos_gcups_vs_seqlen.png)

sw/1d vs sw/2d chip throughput. Both converge to ~30 GCUPS at larger sequences; sw/1d reaches the plateau earlier. sw/2d has a tiny range because its DMEM budget caps seq_len at 192.

### `06_algos_time_vs_seqlen`

![06_algos_time_vs_seqlen](06_algos_time_vs_seqlen.png)

Per-alignment wallclock on log-log axes. Both algorithms show O(n²) scaling; at matched seq_len sw/1d and sw/2d land within a small constant factor.

## Cores-per-group effect

### `07_sw1d_cpg_gcups`

![07_sw1d_cpg_gcups](07_sw1d_cpg_gcups.png)

Chip-wide GCUPS as a function of pipeline length (cpg). Small cpg (1) lets the whole chip run 128 independent pipelines, which wins at short sequences. Long cpg (128) dedicates all 128 cores to one sequence — only pays off at the longest sizes because pipeline-fill overhead dominates otherwise.

### `08_sw1d_cpg_time`

![08_sw1d_cpg_time](08_sw1d_cpg_time.png)

Per-alignment wallclock by cpg. Larger cpg = longer pipeline = each alignment completes faster (more cores work on it in parallel), but the chip runs fewer alignments concurrently. The per-alignment-latency vs chip-throughput trade-off is the whole point of this knob.

## Pod-unique-data effect

### `09_sw1d_pod_unique`

![09_sw1d_pod_unique](09_sw1d_pod_unique.png)

Whether all pods see the same input or each pod gets its own slice. The SW kernels are compute-bound, so their DRAM read pressure is low — we expect no material difference, and the plot confirms that (differences are sub-percent).

### `10_sw2d_pod_unique`

![10_sw2d_pod_unique](10_sw2d_pod_unique.png)

Same question for sw/2d. Again compute-bound and indifferent to whether pods read the same or different slices of DRAM.

## Fast vs simulated high-BW

### `11_sw1d_fast_vs_sim32bw`

![11_sw1d_fast_vs_sim32bw](11_sw1d_fast_vs_sim32bw.png)

sw/1d chip-wide GCUPS, measured at full clock (solid) and projected to a hypothetical 32× DRAM bandwidth machine (dashed). At mid and large seq_len the curves overlap — sw/1d is compute-bound and extra BW doesn't help. The small-seq_len gap where sim32bw > fast reveals that short-sequence runs do hit some BW-related overhead (pipeline-fill reloads, barrier fences) that the 32×-BW projection papers over.

### `12_sw2d_fast_vs_sim32bw`

![12_sw2d_fast_vs_sim32bw](12_sw2d_fast_vs_sim32bw.png)

Same comparison for sw/2d. At seq_len=128 and 192 the curves overlap; at seq_len=32 the sim line is noticeably above, which tells us sw/2d at very short sequences pays a per-rep fixed overhead (barrier + wavefront fill) that would cost proportionally less wallclock on a fast-clock machine with more memory headroom.

### `13_algos_fast_vs_sim32bw`

![13_algos_fast_vs_sim32bw](13_algos_fast_vs_sim32bw.png)

Combined view. Fast and 32× BW lines practically overlap per algorithm — confirming both kernels operate in the compute-bound half of the roofline.

## Roofline

### `14_roofline_fast`

![14_roofline_fast](14_roofline_fast.png)

Chip-wide roofline at the fast core clock. Each dot is one dummy/roofline sweep point; the dashed line is the BW/compute envelope built from the highest measured values.

### `15_roofline_sim32bw`

![15_roofline_sim32bw](15_roofline_sim32bw.png)

Chip-wide roofline under the 32× bandwidth projection (the slow-clock sweep scaled by 32). The BW ceiling rises proportionally, so the ridge point shifts left — most workloads that were BW-bound on fast become compute-bound in this world.

### `16_roofline_overlay`

![16_roofline_overlay](16_roofline_overlay.png)

Both rooflines overlaid. Compute ceilings are nearly identical (the 32× projection preserves per-ALU throughput), while the BW ceiling is 32× higher in the projection — visible as the steeper dashed line shifting up and to the left.

## Summary

### `17_gcups_density`

![17_gcups_density](17_gcups_density.png)

Chip-wide peak GCUPS divided by total pod area (38.875 mm²). cpg=1 narrowly wins on density because it runs 128 independent pipelines; cpg=8 is 98 % of that with simpler per-pipeline resource use. sw/2d matches within a few percent.
